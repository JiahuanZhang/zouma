# ZouMa 产品功能评审报告

> 评审日期：2026-04-16
> 项目：ZouMa - AI 代码评审管理后台

---

## 一、功能设计可优化点

### 1. 任务执行策略透明度不足

- 当前 `simple`/`smart` 策略自动切换，用户无法预知或干预
- **建议**：任务创建时提供策略选择选项，并说明两种策略的差异（文件数阈值、耗时预估）

### 2. 问题列表缺少关键交互

- 无法按文件路径筛选/搜索
- 无法批量操作（忽略、标记已处理）
- 缺少问题定位到源码的能力（点击文件跳转到 IDE）
- **建议**：增加文件搜索、状态标记、导出功能

### 3. 任务重试/恢复机制缺失

- 失败任务只能重新创建，无法一键重试
- 长时间任务中断后无法从断点恢复
- **建议**：增加「重试」按钮，支持增量恢复

### 4. 评审计划触发条件不够灵活

- `interval`/`daily`/`webhook` 三种触发类型，缺少「Git 事件触发」
- 无法关联 Git push/merge 事件自动触发评审
- **建议**：增加 Git webhook 集成（push/PR 事件触发）

### 5. LLM 配置缺少成本预估

- 用户创建任务时无法预知 Token 消耗和费用
- 任务完成后才有统计数据
- **建议**：任务创建前基于文件数/代码量给出费用预估

### 6. 进度可视化层级过深

- 五级粒度（task→phase→batch→agent→tool）信息密度过高
- 普通用户难以快速定位关键信息
- **建议**：提供「简洁模式」开关，隐藏 agent/tool 级别细节

### 7. 文件筛选模式配置分散

- 仓库、计划、任务三处都有 `file_filter_id`，但无继承关系说明
- 用户可能困惑优先级
- **建议**：明确配置继承链，或统一在计划层配置

## 二、功能缺失点

### 1. 缺少评审历史对比

- 无法对比同一仓库不同时间点的评审结果
- 无法追踪问题是否被修复
- **建议**：增加历史评审对比、问题追踪状态（未处理/已修复/持续存在）

### 2. 缺少团队协作功能

- 无用户/角色管理
- 无评审任务分配、评论讨论
- **建议**：增加用户管理、任务认领、评论功能

### 3. 缺少通知机制

- 任务完成/失败无主动通知
- 需要用户手动刷新查看
- **建议**：支持邮件/Webhook/Slack 通知

### 4. 缺少评审规则自定义

- 问题类别固定为 `style/logic/robustness`，无法扩展
- 无法自定义评审规则或添加规则模板
- **建议**：支持自定义问题类别、评审规则配置

### 5. 缺少代码片段展示

- 问题详情只有描述和建议，无原代码片段
- 用户需另行打开文件查看
- **建议**：在问题详情中展示相关代码片段

### 6. 缺少报告导出

- 无法导出 PDF/HTML/Markdown 格式评审报告
- 无法分享给非系统用户
- **建议**：增加报告导出功能

### 7. 缺少仓库状态同步

- 仓库 clone 后无定期 pull/fetch 同步
- 评审基于过期代码
- **建议**：增加仓库同步机制或手动同步按钮

### 8. 缺少评审模板/预设

- 每次任务需手动配置，无保存常用配置的机制
- **建议**：增加任务模板功能，快速复用常用配置

### 9. 缺少 API 文档

- 无 Swagger/OpenAPI 文档
- 外部集成困难
- **建议**：增加 API 文档页面或导出 OpenAPI spec

## 三、优先级建议

| 优先级 | 功能 | 价值 |
|--------|------|------|
| P0 | Git 事件触发评审 | 核心场景，与 DevOps 流程集成 |
| P0 | 任务重试机制 | 降低失败处理成本 |
| P1 | 问题追踪状态 | 闭环评审流程 |
| P1 | 通知机制 | 提升用户体验 |
| P1 | 报告导出 | 满足分享需求 |
| P2 | 成本预估 | 增强可预测性 |
| P2 | 代码片段展示 | 提升问题定位效率 |
| P2 | 评审规则自定义 | 增强灵活性 |

## 五、后续迭代建议

1. Git webhook 集成 → 实现 CI/CD 流程嵌入
2. 任务重试 → 降低运维成本
3. 问题状态追踪 → 闭环评审生命周期

4. 通知机制 → 提升触达效率

7. 团队协作 → 支撑多人场景
8. 自定义规则 → 增强适配性
9. API 文档 → 支撑生态集成
